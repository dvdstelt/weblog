# AGENTS.md

This file provides guidance to AI coding agents when working with code in this repository.

## Commands

```bash
npm run dev -- --host --port 1337        # start dev server at 0.0.0.0:1337 (no search in dev)
npm run build                            # build site + run Pagefind indexer (search requires a full build)
npm run preview -- --host --port 1337    # preview the production build locally
```

There are no automated tests. Verify changes by running `npm run build` and checking for TypeScript/Astro errors.

## Architecture

This is an **Astro 5 static blog** deployed to **Azure Static Web Apps** via GitHub Actions on push to `main`.

### URL structure

Post filenames follow `YYYY-MM-DD-slug.md` and are served at `/YYYY/MM/DD/slug/`. The helper `src/lib/posts.ts` provides `getPostUrl()`, `tagSlug()`, and `readingTime()` which are used throughout. The dynamic route `src/pages/[year]/[month]/[day]/[slug].astro` generates all post pages at build time via `getStaticPaths()`.

### Content

- Posts live in `src/content/posts/` (587 markdown files dating back to 2003).
- The content collection schema is defined in `src/content/config.ts`: required fields are `id` (numeric), `title`, `description`, `pubDate`, `author`, and optional `tags`, `image`, `redirect_from`.
- Tag pages are generated by `src/pages/tag/[tag].astro`; tags are slugified to lowercase-with-hyphens via `tagSlug()`.
- Paginated home/archive pages are at `src/pages/page/[page].astro`.

### Layout hierarchy

`Base.astro` is the root HTML shell (imports `main.scss`, wires up menu, search overlay, scroll-to-top). `Post.astro` and `Page.astro` wrap `Base.astro` for post and static-page content respectively. All interactive behaviour (menu toggle, search overlay, Escape key, scroll button) lives in an inline `<script>` inside `Base.astro`.

### Search

Search uses **Pagefind** (generated after build with `npx pagefind --site dist`). The custom `Search.astro` component loads `pagefind.js` lazily via `new Function('u', 'return import(u)')` to prevent Rollup from trying to statically analyse the generated file. Search is not functional in dev mode - only after a full `npm run build`.

Post content is indexed via `data-pagefind-body` on the article div. Title relevance is boosted with a visually-hidden `data-pagefind-weight="10"` span.

### Styles

SCSS in `src/styles/` follows a layered convention:
- `0-settings/` - variables, mixins, helpers
- `1-tools/` - normalize, reset, grid, animation, syntax highlighting
- `2-base/` - base element styles
- `3-modules/` - component styles (header, footer, search, hero, etc.)
- `4-layouts/` - page-level layout styles

All imported through `src/styles/main.scss`.

### Code snippets from real files

Posts can embed code from real source files under `samples/` instead of inlining copy-pasted snippets. The `remarkCodeRegion` plugin in `astro.config.mjs` (sits alongside `rehypeImageLayout`) finds fenced code blocks whose info string includes `file="path/from/repo-root"` and optionally `region="Name"`, reads the file at build time, extracts the named region, and fills the block. A fence without `file=` is left untouched, so all pre-existing posts are unaffected.

Region marker styles recognised by the plugin:

| Comment style       | Start                   | End                    | Used in                                  |
|---------------------|-------------------------|------------------------|------------------------------------------|
| `#` line / C# native| `# region Name`, `#region Name` | `# endregion`, `#endregion` | C#, shell, Dockerfile, YAML, Python |
| `//` line           | `// region Name`        | `// endregion`         | JS, TS, C, C++, Go, Rust, JSONC          |
| `<!-- -->` block    | `<!-- region Name -->`  | `<!-- endregion -->`   | HTML, XML, `.csproj`                     |
| `/* */` block       | `/* region Name */`     | `/* endregion */`      | CSS, SCSS                                |
| `--` line           | `-- region Name`        | `-- endregion`         | SQL, Lua                                 |

Marker lines are stripped from the rendered snippet and common leading whitespace is dedented. A missing `file=`/`region=` is a hard failure: the plugin writes a `[remark-code-region] ...` error to stderr and calls `process.exit(1)` during `astro build`, so the site build genuinely fails in CI. In `astro dev` it only throws so HMR keeps working.

For each file-imported block the plugin also injects an `<a class="code-source-link">` sibling pointing at the source on GitHub, with a `#Lstart-Lend` fragment for region-based snippets. The wrapper script at the bottom of `Post.astro` pulls that anchor into the same `.code-block-wrapper` as the copy button; styles live in `src/styles/1-tools/_syntax-highlighting.scss`. The GitHub URL prefix and the `main` branch are hardcoded as `GITHUB_BLOB_URL` near the top of `astro.config.mjs` â€” update both if the repo or default branch ever moves.

To add a new code-bearing post: put the source files under `samples/<year>/<post-slug>/` (year of the post's publication date), add any `.csproj` to `samples/Samples.slnx` (`dotnet sln samples/Samples.slnx add samples/<year>/<post-slug>/<Project>/<Project>.csproj`), wrap the interesting parts in region markers, and reference them from the post with `` ```lang file="samples/<year>/<post-slug>/..." region="Name" `` (empty body). The `.NET` samples are built separately by `.github/workflows/samples.yml` with `dotnet build samples/Samples.slnx -warnaserror`, pinned to .NET 10 via `samples/global.json`.

### Diagrams

Architecture and flow diagrams are authored in [D2](https://d2lang.com). There are two ways to embed them:

**Inline fenced block (paste-and-go).** A fenced ` ```d2 ` block in a post is rendered at build time by the `remarkD2` plugin in `astro.config.mjs`, which shells out to the `d2` CLI and inlines the resulting SVG inside a `<figure class="d2-diagram">`. Use this for small diagrams that are intrinsic to a single post. The plugin uses the same defaults as the manual command below (`--sketch --theme=4 -l elk`); it has no per-block override syntax yet, so reach for the file-based workflow when you need a different theme or layout.

**Pre-rendered SVG file.** Source files live at `samples/<year>/<post-slug>/<name>.d2`; the rendered SVG goes to `public/images/<year>/<post-slug>/<name>.svg` so posts can reference it as `/images/<year>/<post-slug>/<name>.svg`. The conventional render command is:

```bash
d2 --sketch --theme=4 -l elk \
   samples/<year>/<post-slug>/<name>.d2 \
   public/images/<year>/<post-slug>/<name>.svg
```

`--sketch` gives the hand-drawn renderer, `--theme=4` is "Cool classics", `-l elk` uses ELK for hierarchical layout (cleaner than the default Dagre on architecture diagrams). The combination is the blog default; deviate only when a specific diagram needs something else, and put the override in the comment at the top of the `.d2` file.

Either way, the `d2` CLI must be on `PATH` (inline blocks render during `astro build`; pre-rendered files render manually before commit). D2 isn't an npm dependency. Install once from the [release page](https://github.com/terrastruct/d2/releases), copy `bin/d2` into a directory on `PATH` (e.g. `~/.local/bin`), and verify with `d2 --version`. CI does not have `d2` installed, so any post using an inline ` ```d2 ` block requires CI to gain `d2` before it can build there; until then, render locally and inspect the output, or use the pre-rendered SVG path for production posts.

### Comments

Giscus (GitHub Discussions-based comments) is embedded via `src/components/GiscusComments.astro` and rendered at the bottom of every post.

### Feed and sitemap

RSS feed is generated by `src/pages/feed.xml.ts`; sitemap by the `@astrojs/sitemap` integration configured in `astro.config.mjs`.

### Deployment

CI/CD is in `.github/workflows/deploy.yml`. It runs `npm ci && npm run build`, then uploads `dist/` to Azure Static Web Apps. PR builds get a preview URL posted as a comment. The `public/staticwebapp.config.json` sets a 404 fallback rewrite to `/404.html`. A second workflow, `.github/workflows/samples.yml`, builds `samples/Samples.slnx` with `dotnet build -warnaserror` on changes to `samples/**` so code referenced by posts can't drift silently.

### Migration scripts

One-off scripts in `scripts/` were used for the Jekyll-to-Astro migration and internal link fixing. They are not part of the regular build.
